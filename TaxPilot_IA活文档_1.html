页面 7 · 首次发现（第一步） 已确认
首次发现 · 三种渲染情况
情况一：全部Clear（理论存在，概率极低）
  • 所有历史交易AI均高置信度判断，无需用户声明
  • 已确认可抵扣总金额大字显示 + 按分类明细展示
  • 预计可减少税款金额
  • 查看明细按钮 → 直接进入过渡页
情况二：部分Clear + 部分模糊（最常见）
  • 已确认金额大字显示 + "另有X笔待你确认"单独列出
  • 按分类列出已确认支出明细
  • 预计可减少税款金额（基于已确认部分）
  • 单一按钮：查看明细 → 进入过渡页 → 主页
  • 不在首次发现页强迫用户做决定，选择权交给用户，进入主页后想改就改，想跳就跳
  • Onboarding流程保持简洁，不因模糊款项而变长
情况三：全部模糊
  • 标题："发现了X笔可能的抵税机会，需要你来确认"
  • 金额显示$???（不给具体数字，给悬念）
  • 说明文字："告诉我每笔支出的用途，我来帮你判断能不能抵税"
  • 开始确认按钮 → 进入主页，省税页待确认分类优先展示
情况四：无历史数据（2026年全新用户）
  • 跳过首次发现页，直接显示欢迎页
  • 文案："你的追踪从今天开始，从现在起每一笔业务支出我们都帮你记录和识别，一分不漏"
  • 开始追踪按钮 → 直接进入主页，不显示过渡页
⚑ 业务逻辑备注（开发时落实）
  • 后端判断路径：全Clear→情况一，部分Clear→情况二，全模糊→情况三，无数据→情况四
  • 情况二用户选"稍后再说"→ 模糊款项保留在待确认队列，主页和省税页随时可处理
  • 历史模糊款项调整入口：日历页（按日期找）+ 省税页待确认分类（按类找）
  • 不单独新增历史调整页，融入日常使用流程

⚑ 业务逻辑备注（开发时落实）
  • 距截止天数 = (2026-04-15 - today()).days，后端直接计算，不需要AI
  • 报税起点为2025年且当前日期早于4月15日 → 显示"直接申报+倒计时"
  • 报税起点为2024年或更早 → 显示"1040-X修正税表"说明
  • 报税起点为2026年 → 不显示历史找回说明，只显示留存承诺

页面 8 · 主页（Tab 1） 已确认
主页
包含信息
  • 四格核心指标：总收入 / 可抵扣支出 / 应税收入 / 预计减少税款
  • 待确认款项数量
  • 待处理交易列表（商户名 + 金额 + 日期 + 状态标签）
功能 / 操作
  • 点击任意交易条目 → 进入支出详情页
  • "添加收入"按钮 → 进入手动收入录入页（K-1 / 现金 / 其他收入）
  • 底部导航栏切换四大 Tab
导航出口
  • → 支出详情页（点击交易条目）
  • → 手动收入录入页（点击添加收入）
  • ↔ 日历 / 省税 / 报告（Tab切换）
⚑ 业务逻辑备注（开发时落实）
  • 季度预估税估算模块已砍掉，不在主页展示
  • 季度税务截止日仅通过推送通知提醒用户，不在主页显示估算数字
  • 四格指标计算链：总收入 - 可抵扣支出 = 应税收入 × 税率 = 预计税款，与季度预估税无关，只记录已发生收入
  • 总收入 = 今年Plaid已识别入账收入 + 用户手动录入收入，所有数字基于今年数据，时间维度统一
  • 所有数字均为年度估算，随时更新，界面加注"基于今年已入账收入估算 · 随时更新"
  • 季度缴税≠季度计税：IRS按年度计税，季度只是预付机制，产品层面只显示年度汇总
  • 边界情况：今年收入 < 今年可抵扣支出 → 应税收入显示$0，预计税款显示$0，加注"今年支出超过收入，本年度暂无应税收入"
  • 没有收入变量，省税数字无法成立，四格指标必须完整展示
  • 核心聚焦：帮用户找回可抵扣支出，告诉他能少交多少税

页面 11 · 报告（Tab 4） 已确认
报告
包含信息
  • 年度切换选项（今年 / 去年）
  • 第一层（默认展开）：四格核心指标——总收入 / 可抵扣支出 / 应税收入 / 预计减少税款

  • 第二层（默认收起，点击展开）：按分类列出可抵扣支出明细，每条显示商户名 + 金额 + 抵扣比例 + IRS依据摘要

  • 点击单笔抵扣条目 → 进入支出详情页

  • 信息三层分级：总览→分类明细→单笔详情，用户一次只看一个层级，视觉不冗余
  • 抵扣明细默认收起，用户主动展开才显示，保持页面干净
  • 无独立抵扣明细页和发送CPA页，所有操作在报告页内完成
  • 发送给CPA的报告包含：收入汇总 / 每笔抵扣明细 / 银行交易ID / 用户声明原文 / IRS依据 / 税务计算过程
  • 自行报税指南按Schedule组织输出：Schedule C各行（广告/软件/差旅/餐饮×50%/家庭办公室等）/ Schedule SE自雇净收入 / Schedule E（K-1收入如有）/ Schedule 1自雇税抵扣
  • 每行标注对应税表行号，用户打开TurboTax或FreeTaxUSA直接对号入座填数字，无需理解税法
  • 不用CPA也能完整报税是产品核心差异化之一：$240/年TaxPilot vs $500-$2,000/年CPA
页面 12 · 支出详情页（四个入口共用） 已确认
支出详情页
包含信息
  • 商户名称 + 金额 + 日期 + 来源（Plaid银行账户 / 手动录入）
  • 税务判断区块：性质 / 抵扣比例 / 可抵扣金额 / 判断来源（AI自动识别 / 用户声明）/ 状态（已确认 / 待确认）
  • 三层凭证链：① 银行原始记录（交易ID）/ ② 判断来源（AI识别逻辑 或 用户声明原文）/ ③ IRS依据（Publication编号 + 条款描述）
功能 / 操作
  • 待确认状态：AI返回多分支预计算结果，每个分支直接显示分类名称+可抵扣金额+省税数字+IRS依据，用户选择后立即确认，无需二次AI调用
  • 已确认状态·修改声明：自由文本输入框，用户描述新用途 → 标注"已修改·未提交"橙色状态，不立即触发AI
  • 手动录入状态：填写商户名称 + 金额 + 日期 + 用途声明 + 生成卡片 → 标注待提交
  • 底部浮出"提交X条修改"按钮（有未提交修改时出现）→ 一次性打包AI处理所有修改声明
  • 离开页面时若有未提交修改 → 提示"你有X条修改未提交，是否现在提交？"
  • 分支选择和批量提交是两套独立机制：分支选择无需提交按钮直接确认，修改声明才需要批量提交
导航出口
  • ← 返回来源页（主页 / 日历 / 省税 / 报告，保留各自入口上下文）
⚑ 业务逻辑备注（开发时落实）
  • 四个入口（主页/日历/省税/报告）共用同一个组件，返回时回到各自来源页
  • 用户声明内容由用户自己负责，产品忠实记录并生成凭证链
  • 任何已确认记录随时可修改，没有锁死状态
  • 修改采用批量提交机制：单次修改只标注状态，不触发AI；点击"提交X条修改"后统一打包AI处理
  • 批量提交减少AI调用次数，节约成本，提升响应效率
  • 提交后所有相关数字统一更新

页面 14 · 账户重新认证流程 已确认
账户重新认证（Plaid Re-auth）
触发场景
  • 银行要求定期重新授权（通常每90天到1年一次，取决于银行）
  • 只影响需要重新认证的那张卡，其他账户正常运行
包含信息
  • 推送通知：精确到具体账户名（如"Chase Checking需要重新验证"）
  • 主页Banner：橙色警告，精确显示哪张卡出问题，其他账户显示"正常同步中"
  • 设置页账户列表：每张卡独立状态，出问题的显示橙色⚠标识
  • 重新认证页：说明原因（银行安全机制）+ 承诺历史记录不丢失
  • 恢复成功页：绿色确认，告知数据已恢复
功能 / 操作
  • 点击推送通知 → 直接进入重新认证页
  • 点击主页Banner"重新连接" → 进入重新认证页
  • 点击设置页该账户"重新连接" → 进入重新认证页
  • 重新认证页点击"通过Plaid重新连接" → 唤起Plaid Link Update Mode
  • 认证成功 → 显示恢复成功页 → 返回主页
导航出口
  • → Plaid Link Update Mode（重新认证）
  • → 主页（认证成功后）
⚑ 业务逻辑备注（开发时落实）
  • 监听Plaid的PENDING_EXPIRATION webhook → 提前提醒用户
  • 监听ITEM_LOGIN_REQUIRED错误 → 触发重新认证流程
  • 多卡情况：每张卡独立状态，互不影响
  • 重新认证后历史数据完整保留，不删除不重置
  • 每个步骤强调"历史记录不丢失"，降低用户焦虑和流失
  • 这是留存的关键节点，用户忽略提示会导致数据断档，影响省税准确性

⚑ 业务逻辑备注（开发时落实）
  • 修改实体类型可模拟不同税务结构下的数字，用户看完可随时改回
  • 账户连接有两种操作：① 暂停同步——停止拉取新交易，历史数据和连接状态保留，随时可恢复；② 断开连接——彻底撤销Plaid授权，但历史交易数据仍保留在TaxPilot，不删除
  • 二次确认弹窗必须清楚说明"断开不等于数据消失"
  • 断开账户为Plaid合规要求，必须提供此入口
  • 删除账户及数据为不可逆操作，需二次确认
