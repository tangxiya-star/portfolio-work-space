The Store — Global Memory

Imagine a whiteboard that every screen in the app can read and write at any time. That is Zustand. The store holds exactly three categories of information — and nothing more.

CODE 
import { create } from 'zustand'

interface TaxStore {
  userId: string | null
  accessToken: string | null
  pendingSubmitIds: string[]
  addPendingSubmit: (id: string) => void
  removePendingSubmit: (id: string) => void
  clearPendingSubmit: () => void
  onboardingComplete: boolean
  setOnboardingComplete: (v: boolean) => void
  setAuth: (userId: string, token: string) => void
  clearAuth: () => void
}

export const useTaxStore = create<TaxStore>((set) => ({
  userId: null,
  accessToken: null,
  pendingSubmitIds: [],
  addPendingSubmit: (id) =>
    set((state) => ({
      pendingSubmitIds: state.pendingSubmitIds.includes(id)
        ? state.pendingSubmitIds
        : [...state.pendingSubmitIds, id],
    })),
}))
PLAIN ENGLISH

This is the app's shared memory. Any screen can read or change these values without passing them around manually.

userId = who is currently logged in. It starts as null (nobody) and gets set when the user signs in.

pendingSubmitIds = the list of expenses waiting for batch approval. When you tap "Submit All," this list is what gets sent.

addPendingSubmit is careful — it checks the list first and refuses to add a duplicate. No expense gets submitted twice.

create() sets everything up in one place so all 23 screens draw from the same single source of truth.

⚠️
The Rule: Only 3 things live in Zustand — auth state, batch submit queue, and onboarding progress. Transactions and tax data go through React Query, not the store.

Hooks — Logic with a Name

A hook is a function that packages up a piece of logic and gives it a name. Instead of writing the same fetch-and-cache code in every screen, you write it once in a hook and call it anywhere.

CODE 
import { useQuery } from '@tanstack/react-query'
import type { Transaction } from '../types/transaction'

import { mockTransactions } from '../mocks/data'

export function useTransactions() {
  return useQuery({
    queryKey: ['transactions', 'home'],
    queryFn: async () => {
      return mockTransactions
    },
    staleTime: 30 * 1000,
  })
}
PLAIN ENGLISH

useTransactions() asks React Query to fetch the transaction list. Any screen that calls this hook gets the same data — no duplicate requests.

queryKey: ['transactions', 'home'] is the address in the cache. If two screens use the same key, they share one copy of the data.

queryFn is what actually runs to get the data. Right now it returns mock data — when the backend is live, only this one line changes.

staleTime: 30 * 1000 means if you asked in the last 30 seconds, use the cached answer — no network call needed.

CODE — hooks/useTaxSummary.ts 

import { useQuery } from '@tanstack/react-query'
import { api } from '../services/api'
import type { TaxSummary } from '../types/tax'

import { mockTaxSummary } from '../mocks/data'

export function useTaxSummary(year: number) {
  return useQuery({
    queryKey: ['tax-summary', year],
    queryFn: async () => {
      // return api.get<TaxSummary>(`/tax/summary?year=${year}`)
      return mockTaxSummary
    },
    staleTime: 30 * 1000,
    refetchOnWindowFocus: true,
  })
}
PLAIN ENGLISH

Lines 1–3: Import the useQuery hook and TypeScript type definitions for TaxSummary data.

Line 5: Import fake data for development — this import gets removed entirely when the backend is ready.

Lines 7–8: This hook accepts a year (2024, 2025, 2026) as its input argument.

Line 9: Start a React Query data-fetching request managed by the cache.

Line 10: The cache address — "tax summary for this specific year." Change the year, get a different slot.

Lines 11–13: The actual fetcher — currently returns fake data. The real API call sits above it, commented out.

Line 15: Consider data fresh for 30 seconds. No network re-fetch happens during this window.

Line 16: Re-fetch automatically when the user comes back to the app from the background.

💡
Engineering elegance. The screen component does not know or care whether data came from mock or real API — zero UI changes needed when the backend goes live. Swap one line, ship to production.

The API Service — One Gate for All Traffic

Every network request in TaxPilot passes through a single file. One gate. One rule enforcer.

CODE — services/api.ts 

const BASE_URL = process.env.EXPO_PUBLIC_API_URL
  ?? 'http://localhost:8000/api'

class APIError extends Error {
  code: string
  constructor(code: string, message: string) {
    super(message)
    this.code = code
    this.name = 'APIError'
  }
}

async function request<T>(
  endpoint: string,
  options?: RequestInit
): Promise<T> {
  const token = await storage.getToken()
  const response = await fetch(`${BASE_URL}${endpoint}`, {
    headers: {
      'Content-Type': 'application/json',
      ...(token ? { Authorization: `Bearer ${token}` } : {}),
    },
    ...options,
  })

  if (!response.ok) {
    const error = await response.json()
    throw new APIError(error.error.code, error.error.message)
  }

  return response.json()
}

export const api = {
  get: <T>(endpoint: string) =>
    request<T>(endpoint),
  post: <T>(endpoint: string, body: unknown) =>
    request<T>(endpoint, { method: 'POST', body: JSON.stringify(body) }),
}
PLAIN ENGLISH

BASE_URL — The server address. Reads from an environment variable; falls back to localhost during development.

APIError — A custom error type that carries a machine-readable code alongside the human message. Screens can switch on the code to show specific error UI.

request() — The universal template. It grabs the stored auth token, attaches it to every request as a Bearer header, calls fetch, and throws a typed APIError if the server responds with an error status.

api.get / api.post — Clean, one-line wrappers the rest of the app calls. The caller never sees fetch() directly — they just call api.get('/tax/summary?year=2026').

🚪
One gate. No fetch() appears anywhere else in the app — only here. If the API URL or auth method ever changes, there is exactly one file to update.

CODE — constants/theme.ts (colour section) 

const brand = {
  ink: '#0D0D0D',
  newsprint: '#E3DFD5',
  emerald: '#22C55F',
  white: '#FFFFFF',
} as const

export const theme = {
  colors: {
    ...brand,
    primary: brand.ink,
    accent: brand.emerald,
    background: brand.newsprint,
    surface: brand.white,
    textPrimary: brand.ink,
    textSecondary: hexToRgba(brand.ink, 0.6),
    textTertiary: hexToRgba(brand.ink, 0.5),
    border: hexToRgba(brand.ink, 0.1),
    green: brand.emerald,
    greenBg: hexToRgba(brand.emerald, 0.1),
    amber: '#C2410C',
    amberBg: 'rgba(194, 65, 12, 0.1)',
  },
PLAIN ENGLISH

brand — the 4 raw colour values. The as const tells TypeScript these strings are exact literals, not just any string. These are never referenced directly in components.

theme.colors builds semantic names on top of the primitives. textPrimary means ink. accent means emerald. This indirection is the entire point of the system.

hexToRgba() creates semi-transparent versions. textSecondary is ink at 60% opacity — a softer label colour without introducing a new hex value.

amber is the one exception — a status colour with no primitive alias. It still lives in theme.ts so it has a single source of truth.

⚠️
CLAUDE.md rule: Never hardcode hex values in components. If AI writes color: '#0D0D0D' in a screen, flag it immediately — it should be theme.colors.ink.

StyleSheet.create — Styles as Objects

React Native has no CSS files. Instead, styles are JavaScript objects — but the pattern maps directly to CSS concepts you already know.

CODE — PendingPill StyleSheet block 

const pendingStyles = StyleSheet.create({
  pill: {
    flexDirection: 'row',
    alignItems: 'center',
    alignSelf: 'flex-start',
    backgroundColor: theme.colors.surface,
    paddingHorizontal: theme.spacing.lg,
    paddingVertical: theme.spacing.sm,
    minHeight: 44,
    borderRadius: theme.radius.round,
    gap: theme.spacing.sm,
  },
  pillPressed: {
    backgroundColor: theme.colors.ink,
  },
  text: {
    fontWeight: '600',
    color: theme.colors.textPrimary,
  },
  textPressed: {
    color: theme.colors.white,
  },
})
PLAIN ENGLISH

StyleSheet.create() is like writing CSS classes. Each key (pill, text) is a named style object you apply to a component — exactly like a CSS class name.

flexDirection: row = CSS display: flex; flex-direction: row. paddingHorizontal = CSS padding-left + padding-right combined into one shorthand.

minHeight: 44 is an Apple HIG requirement. Touch targets must be at least 44×44 points. This is an accessibility rule, not a preference.

pillPressed / textPressed are the pressed-state overrides. The component applies both style arrays simultaneously when the button is active: style={'{'}[styles.pill, isPressed && styles.pillPressed]{'}'}.

👆
minHeight: 44 is not optional. Apple Human Interface Guidelines require touch targets to be at least 44×44 points. Too small means users miss taps — especially for people with motor impairments. This is an accessibility requirement baked into every interactive component in TaxPilot.

The Hook Extraction Rule

Screen components are dumb on purpose. All the intelligence lives in hooks. Here is what that looks like in practice.

CODE — hooks/useOnboardingStep1.ts 

export function useOnboardingStep1() {
  const [selectedIncome, setSelectedIncome] = useState<string[]>([])
  const [selectedEntity, setSelectedEntity] = useState<string>('')
  const [selectedOccupation, setSelectedOccupation] = useState<string>('')

  const isComplete =
    selectedIncome.length > 0 &&
    !!selectedEntity &&
    !!selectedOccupation

  const handleContinue = async () => {
    // save to store, navigate to step2
  }

  return {
    selectedIncome, selectedEntity, selectedOccupation,
    isComplete, handleContinue,
    setSelectedIncome, setSelectedEntity, setSelectedOccupation,
  }
}
PLAIN ENGLISH

Line 1: This hook manages all the state for the first onboarding screen.

Lines 2–4: Three pieces of state — what income types the user selected, their entity type, and their occupation.

Lines 6–9: isComplete is true only when all three fields have a value — this controls whether the Continue button is enabled.

Lines 11–13: handleContinue saves selections and navigates to step 2.

Lines 15–24: Return everything the screen component needs — it just receives and uses these values.

What Screen Files Do

  • → Render JSX elements
  • → Apply styles from StyleSheet
  • → Import and call hooks
  • → Show/hide things based on hook data

What Hooks Do

  • → Manage useState and state updates
  • → Call services (API calls)
  • → Run business logic (validation, calculations)
  • → Navigate between screens
📌
Rule from CLAUDE.md: No business logic in components — extract to hooks/. When AI puts logic in a screen component, ask it to move it to a hook.

TypeScript — The Safety Net

TypeScript acts like airport security: every value is checked before it passes through. Nothing ships broken.

CODE — types/transaction.ts 

export type TransactionStatus = 'pending' | 'confirmed' | 'excluded'

export type TransactionCategory =
  | 'software'
  | 'travel'
  | 'office'
  | 'meals'
  | 'personal'
  | 'income'
  | 'other'

export interface Transaction {
  id: string
  merchant_name: string
  amount: number
  date: string
  status: TransactionStatus
  category: TransactionCategory | null
  account_id: string
  account_mask: string
}
PLAIN ENGLISH

Line 1: TransactionStatus can ONLY be one of these three values — anything else refuses to compile.

Lines 3–10: TransactionCategory is a fixed list — writing 'food' anywhere in the codebase immediately causes a TypeScript error.

Lines 12–20: A Transaction is a contract — every transaction object must have exactly these fields with exactly these types.

Line 15: amount is always a number — never a string like "$54.99".

Line 17: category can be null (uncategorized) or one of the valid categories — both are allowed.

Without TypeScript

status could be 'approved', 'rejected', or 'done' — each developer uses different values, logic silently breaks, bugs reach users.

With TypeScript Strict

Write 'approved' anywhere and the editor underlines it in red immediately — before the app even runs.

🛡️
Rule from CLAUDE.md: No any type — strict TypeScript everywhere. If AI uses any, ask it to replace it with the proper type from types/. The types folder is the app's contract with itself.

Adding a Feature — The Full Loop

Every new feature in TaxPilot follows the same five-step pattern. Here is how to add a Notes field to a transaction — from type to screen.

1

📝 Add the type

Add notes?: string to the Transaction interface in types/transaction.ts. This is the contract — everything else flows from here.

2

🌐 Add the service call

Add patch('/transactions/:id', { notes }) in services/api.ts. All network calls live here — never directly in a component or hook.

3

🪝 Create the hook

Create hooks/useUpdateNote.ts with a React Query mutation that calls the service and invalidates the cache on success.

4

📱 Update the screen

Render a text input in app/detail.tsx and call useUpdateNote. The screen stays thin — no logic, just UI and a hook call.

5

🎨 Style it

Use only theme.colors and typography tokens — never add new hex values. The design system handles consistency automatically.
