MODULE 5 / SCREEN 1

File = Route

Adding a new screen to TaxPilot takes 30 seconds — create a file in the right folder and it appears automatically. Here is the magic behind that.

📁 app/ All screens — filename = route
📄 index.tsx → / (root redirect to /intro)
📄 detail.tsx → /detail (transaction modal)
📁 (auth)/ route group — does NOT appear in URL
📄 intro.tsx → /intro
📄 welcome.tsx → /welcome
📁 onboarding/
📄 step1.tsx → /onboarding/step1
📄 step2.tsx → /onboarding/step2
📄 step3.tsx → /onboarding/step3
📄 connect.tsx → /connect
📄 loading.tsx → /loading
📄 analyzing.tsx → /analyzing
📁 discovery/
📄 index.tsx → /discovery
📄 case1.tsx → /discovery/case1
📁 (tabs)/ route group — becomes the tab bar
📄 index.tsx → Home tab
📄 calendar.tsx → Calendar tab
📄 savings.tsx → Savings tab
📄 report.tsx → Report tab

Parentheses in folder names like (auth) and (tabs) create route groups — they organise files without adding a URL segment. The folder name is invisible in the final URL.

💡
Aha! To add a new tab, create a file in app/(tabs)/. The router finds it automatically — no imports, no config changes. This is called file-based routing.
CODE 

<Tabs screenOptions={{ tabBarStyle: { backgroundColor: theme.colors.background } }}>
  <Tabs.Screen name="index"    options={{ title: 'Home',     tabBarIcon: ... }} />
  <Tabs.Screen name="calendar" options={{ title: 'Calendar', tabBarIcon: ... }} />
  <Tabs.Screen name="savings"  options={{ title: 'Savings',  tabBarIcon: ... }} />
  <Tabs.Screen name="report"   options={{ title: 'Report',   tabBarIcon: ... }} />
</Tabs>
PLAIN ENGLISH

name="index" matches the file app/(tabs)/index.tsx automatically.

Add a Tabs.Screen for any new file you create — that is the only registration required.

The tab bar renders in the order the screens appear here, top to bottom.

MODULE 5 / SCREEN 2

The Hook Extraction Rule

Screen components are dumb on purpose. All the intelligence lives in hooks. Here is what that looks like in practice.

CODE — hooks/useOnboardingStep1.ts 

export function useOnboardingStep1() {
  const [selectedIncome, setSelectedIncome] = useState<string[]>([])
  const [selectedEntity, setSelectedEntity] = useState<string>('')
  const [selectedOccupation, setSelectedOccupation] = useState<string>('')

  const isComplete =
    selectedIncome.length > 0 &&
    !!selectedEntity &&
    !!selectedOccupation

  const handleContinue = async () => {
    // save to store, navigate to step2
  }

  return {
    selectedIncome, selectedEntity, selectedOccupation,
    isComplete, handleContinue,
    setSelectedIncome, setSelectedEntity, setSelectedOccupation,
  }
}
PLAIN ENGLISH

Line 1: This hook manages all the state for the first onboarding screen.

Lines 2–4: Three pieces of state — what income types the user selected, their entity type, and their occupation.

Lines 6–9: isComplete is true only when all three fields have a value — this controls whether the Continue button is enabled.

Lines 11–13: handleContinue saves selections and navigates to step 2.

Lines 15–24: Return everything the screen component needs — it just receives and uses these values.

What Screen Files Do

  • → Render JSX elements
  • → Apply styles from StyleSheet
  • → Import and call hooks
  • → Show/hide things based on hook data

What Hooks Do

  • → Manage useState and state updates
  • → Call services (API calls)
  • → Run business logic (validation, calculations)
  • → Navigate between screens
📌
Rule from CLAUDE.md: No business logic in components — extract to hooks/. When AI puts logic in a screen component, ask it to move it to a hook.

MODULE 5 / SCREEN 3

TypeScript — The Safety Net

TypeScript acts like airport security: every value is checked before it passes through. Nothing ships broken.

CODE — types/transaction.ts 

export type TransactionStatus = 'pending' | 'confirmed' | 'excluded'

export type TransactionCategory =
  | 'software'
  | 'travel'
  | 'office'
  | 'meals'
  | 'personal'
  | 'income'
  | 'other'

export interface Transaction {
  id: string
  merchant_name: string
  amount: number
  date: string
  status: TransactionStatus
  category: TransactionCategory | null
  account_id: string
  account_mask: string
}
PLAIN ENGLISH

Line 1: TransactionStatus can ONLY be one of these three values — anything else refuses to compile.

Lines 3–10: TransactionCategory is a fixed list — writing 'food' anywhere in the codebase immediately causes a TypeScript error.

Lines 12–20: A Transaction is a contract — every transaction object must have exactly these fields with exactly these types.

Line 15: amount is always a number — never a string like "$54.99".

Line 17: category can be null (uncategorized) or one of the valid categories — both are allowed.

Without TypeScript

status could be 'approved', 'rejected', or 'done' — each developer uses different values, logic silently breaks, bugs reach users.

With TypeScript Strict

Write 'approved' anywhere and the editor underlines it in red immediately — before the app even runs.

🛡️
Rule from CLAUDE.md: No any type — strict TypeScript everywhere. If AI uses any, ask it to replace it with the proper type from types/. The types folder is the app's contract with itself.

MODULE 5 / SCREEN 4

Adding a Feature — The Full Loop

Every new feature in TaxPilot follows the same five-step pattern. Here is how to add a Notes field to a transaction — from type to screen.

1

📝 Add the type

Add notes?: string to the Transaction interface in types/transaction.ts. This is the contract — everything else flows from here.

2

🌐 Add the service call

Add patch('/transactions/:id', { notes }) in services/api.ts. All network calls live here — never directly in a component or hook.

3

🪝 Create the hook

Create hooks/useUpdateNote.ts with a React Query mutation that calls the service and invalidates the cache on success.

4

📱 Update the screen

Render a text input in app/detail.tsx and call useUpdateNote. The screen stays thin — no logic, just UI and a hook call.

5

🎨 Style it

Use only theme.colors and typography tokens — never add new hex values. The design system handles consistency automatically.
