Meet the Cast

TaxPilot has 23 files in app/ — but only 4 types of characters doing the real work. Learn the cast and any file makes sense.

The 4 Archetypes

Think of the codebase like a TV production crew. Some people are on camera, some run the memory department, some do research. Every file in TaxPilot belongs to one of these four roles.

📱

Screen

What the user sees and touches. A Screen just arranges things on the page — it contains no logic of its own.

app/(tabs)/index.tsx
🪝

Hook

Logic extracted from screens so it can be reused and tested independently. Hooks fetch, filter, and transform data.

hooks/useTransactions.ts
🗄️

Store

Global memory that any screen can read without passing props. It survives navigation — your login stays yours.

store/taxStore.ts
🌐

Service

Everything that talks to the outside world. All network requests flow through one place so they are easy to swap when the backend goes live.

services/api.ts

The Store — Global Memory

Imagine a whiteboard that every screen in the app can read and write at any time. That is Zustand. The store holds exactly three categories of information — and nothing more.

CODE 
import { create } from 'zustand'

interface TaxStore {
  userId: string | null
  accessToken: string | null
  pendingSubmitIds: string[]
  addPendingSubmit: (id: string) => void
  removePendingSubmit: (id: string) => void
  clearPendingSubmit: () => void
  onboardingComplete: boolean
  setOnboardingComplete: (v: boolean) => void
  setAuth: (userId: string, token: string) => void
  clearAuth: () => void
}

export const useTaxStore = create<TaxStore>((set) => ({
  userId: null,
  accessToken: null,
  pendingSubmitIds: [],
  addPendingSubmit: (id) =>
    set((state) => ({
      pendingSubmitIds: state.pendingSubmitIds.includes(id)
        ? state.pendingSubmitIds
        : [...state.pendingSubmitIds, id],
    })),
}))
PLAIN ENGLISH

This is the app's shared memory. Any screen can read or change these values without passing them around manually.

userId = who is currently logged in. It starts as null (nobody) and gets set when the user signs in.

pendingSubmitIds = the list of expenses waiting for batch approval. When you tap "Submit All," this list is what gets sent.

addPendingSubmit is careful — it checks the list first and refuses to add a duplicate. No expense gets submitted twice.

create() sets everything up in one place so all 23 screens draw from the same single source of truth.

⚠️
The Rule: Only 3 things live in Zustand — auth state, batch submit queue, and onboarding progress. Transactions and tax data go through React Query, not the store.

Hooks — Logic with a Name

A hook is a function that packages up a piece of logic and gives it a name. Instead of writing the same fetch-and-cache code in every screen, you write it once in a hook and call it anywhere.

CODE 
import { useQuery } from '@tanstack/react-query'
import type { Transaction } from '../types/transaction'

import { mockTransactions } from '../mocks/data'

export function useTransactions() {
  return useQuery({
    queryKey: ['transactions', 'home'],
    queryFn: async () => {
      return mockTransactions
    },
    staleTime: 30 * 1000,
  })
}
PLAIN ENGLISH

useTransactions() asks React Query to fetch the transaction list. Any screen that calls this hook gets the same data — no duplicate requests.

queryKey: ['transactions', 'home'] is the address in the cache. If two screens use the same key, they share one copy of the data.

queryFn is what actually runs to get the data. Right now it returns mock data — when the backend is live, only this one line changes.

staleTime: 30 * 1000 means if you asked in the last 30 seconds, use the cached answer — no network call needed.

The Root Layout — The Stage Manager

Every actor needs a stage. The root layout (app/_layout.tsx) wraps the entire app in two invisible providers that make everything else possible.

CODE 
  return (
    <View style={{ flex: 1 }} onLayout={onLayoutRootView}>
      <QueryClientProvider client={queryClient}>
        <StatusBar style="dark" />
        <Stack screenOptions={{ headerShown: false }}>
          <Stack.Screen name="(auth)" />
          <Stack.Screen name="(tabs)" />
          <Stack.Screen name="detail" />
        </Stack>
      </QueryClientProvider>
    </View>
  )
PLAIN ENGLISH

QueryClientProvider wraps the whole app so every hook can use React Query's cache. Remove this and every useQuery call breaks instantly.

This is the provider pattern in action. You configure the cache once at the top, and every screen below inherits it automatically.

Stack navigator defines three zones: auth screens (welcome, onboarding), tab screens (home, calendar, savings), and the detail modal. All 23 screens live inside one of these three groups.

💡
Mental model: QueryClientProvider is the building's electricity supply. Stack is the floor plan. Every room (screen) gets power automatically — but only because the supply is connected at the root.
