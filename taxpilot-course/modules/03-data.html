How Data Travels

You tap a transaction row — TaxPilot shows details instantly with no loading spinner. How? The data was already cached before you tapped.

Think of TaxPilot as a postal sorting centre. React Query is the sorting system. Every data request has a mailbox address called a queryKey. Screens are the recipients.

CODE — hooks/useTaxSummary.ts 

import { useQuery } from '@tanstack/react-query'
import { api } from '../services/api'
import type { TaxSummary } from '../types/tax'

import { mockTaxSummary } from '../mocks/data'

export function useTaxSummary(year: number) {
  return useQuery({
    queryKey: ['tax-summary', year],
    queryFn: async () => {
      // return api.get<TaxSummary>(`/tax/summary?year=${year}`)
      return mockTaxSummary
    },
    staleTime: 30 * 1000,
    refetchOnWindowFocus: true,
  })
}
PLAIN ENGLISH

Lines 1–3: Import the useQuery hook and TypeScript type definitions for TaxSummary data.

Line 5: Import fake data for development — this import gets removed entirely when the backend is ready.

Lines 7–8: This hook accepts a year (2024, 2025, 2026) as its input argument.

Line 9: Start a React Query data-fetching request managed by the cache.

Line 10: The cache address — "tax summary for this specific year." Change the year, get a different slot.

Lines 11–13: The actual fetcher — currently returns fake data. The real API call sits above it, commented out.

Line 15: Consider data fresh for 30 seconds. No network re-fetch happens during this window.

Line 16: Re-fetch automatically when the user comes back to the app from the background.

💡
Engineering elegance. The screen component does not know or care whether data came from mock or real API — zero UI changes needed when the backend goes live. Swap one line, ship to production.

MODULE 3 / SCREEN 3

The API Service — One Gate for All Traffic

Every network request in TaxPilot passes through a single file. One gate. One rule enforcer.

CODE — services/api.ts 

const BASE_URL = process.env.EXPO_PUBLIC_API_URL
  ?? 'http://localhost:8000/api'

class APIError extends Error {
  code: string
  constructor(code: string, message: string) {
    super(message)
    this.code = code
    this.name = 'APIError'
  }
}

async function request<T>(
  endpoint: string,
  options?: RequestInit
): Promise<T> {
  const token = await storage.getToken()
  const response = await fetch(`${BASE_URL}${endpoint}`, {
    headers: {
      'Content-Type': 'application/json',
      ...(token ? { Authorization: `Bearer ${token}` } : {}),
    },
    ...options,
  })

  if (!response.ok) {
    const error = await response.json()
    throw new APIError(error.error.code, error.error.message)
  }

  return response.json()
}

export const api = {
  get: <T>(endpoint: string) =>
    request<T>(endpoint),
  post: <T>(endpoint: string, body: unknown) =>
    request<T>(endpoint, { method: 'POST', body: JSON.stringify(body) }),
}
PLAIN ENGLISH

BASE_URL — The server address. Reads from an environment variable; falls back to localhost during development.

APIError — A custom error type that carries a machine-readable code alongside the human message. Screens can switch on the code to show specific error UI.

request() — The universal template. It grabs the stored auth token, attaches it to every request as a Bearer header, calls fetch, and throws a typed APIError if the server responds with an error status.

api.get / api.post — Clean, one-line wrappers the rest of the app calls. The caller never sees fetch() directly — they just call api.get('/tax/summary?year=2026').

🚪
One gate. No fetch() appears anywhere else in the app — only here. If the API URL or auth method ever changes, there is exactly one file to update.
