Decision 02

Two AI tools. One human in the middle.

Shipping ~10K lines solo, in six narratively distinct commits, took leverage. The skill isn't I used AI. It's knowing which AI for which stage — and where to override it.

variant.com

For generation

Picked over v0 / Lovable specifically because its output reads as designed, not generated. The mainstream AI UI tools produce instantly-recognizable patterns I'd have to spend cycles fighting; variant gave a scaffold I could build on. Tool selection itself is a design decision.

Claude Code

For iteration & system maintenance

Bound by CLAUDE.md rules and .claude/memory/ persistent feedback. Each session inherits the design system as a hard constraint, so the AI extends the codebase without reintroducing the entropy variant started with.

Claude Code (constrained)

AI for the parts that benefit from speed and breadth. Human for the parts that need judgment — translation across platforms, refactoring toward a system, writing the rules that lock the next AI session in.

The rules the AI is bound by

# CLAUDE.md — excerpt

## Critical Rules
1. No direct fetch in components — all API calls go through services/api.ts
2. No server data in Zustand — managed by React Query
3. No `any` type — strict TypeScript everywhere
4. No business logic in components — extract to hooks/
5. Colors from theme only — never hardcode hex
6. Green is #22C55F — the only green allowed anywhere
Decision 03

Tax apps live or die on trust. Here's how I designed for it.

Trust isn't a copywriting problem. It's designed across the surface, in moments most users never consciously notice. Six small ones from TaxPilot.

01
Plaid gets the spotlight
On the bank-connect screen, the Plaid logo is prominent. Borrowed authority — leveraged on purpose.
02
Reauth is a banner, not a wall
A blocking modal says "the app is broken." A persistent banner says "this is recoverable, we're handling it."
03
Reconnect is a full flow
Errors in finance apps are trust events. Treating them as dead-ends erodes confidence; designing recovery preserves it.
04
The paywall waits
Appears only after the user sees their first analyzed transactions. Up front says "pay to find out;" after value says "this works — keep going."
05
Analyzing narrates itself
A blank spinner with the user's bank data is, in a tax context, slightly uncomfortable. Narration is reassurance.
06
Numbers always align
Misaligned columns subconsciously read as imprecise. Aligned columns read as audited.

What I'd ship differently next time.

No portfolio is honest without this part. Five things I'd reorder, redesign, or delete if I started again on Monday.

  1. 01
    [Build the design system before the first UI commit, not refactor into it at commit 4. The "design system enforcement" commit added 1,860 lines and removed 1,207 — most of that should never have been there.]
  2. 02
    [Set up Storybook (or Expo's equivalent) on day one. The component playground above is exactly what I wished I'd had during build.]
  3. 03
    [User-test the loading → analyzing split before assuming it works. I believe the perceived-performance argument, but I haven't measured it.]
  4. 04
    [Treat the variant.com → RN translation as a script, not a manual chore. Repeated work is automation work.]
  5. 05
    [Write CLAUDE.md before writing any code. Constraints retroactively applied are 10× more expensive than constraints set up front.]

These are starter drafts. Replace with your actual hindsight — the more specific and a little uncomfortable, the better.
